Problem
왜 만들었나

동물병원은 대부분 전화 예약·종이 문진으로 돌아가고, 보호자는 “지금 병원에 가야 할 응급인지”를 스스로 판단하기 어렵습니다. 수의사는 진료와 행정을 동시에 떠안고, 상담·문진·예약·진료·처방·경과 기록이 서로 끊겨 있습니다.

PAIN

흩어진 진료 여정

  • 상담 → 예약 → 진료 → 처방 → 경과가 각각 다른 채널·종이로 단절
  • 보호자는 응급도 판단이 어렵고, 수의사는 사전 정보 없이 진료 시작
SOLUTION

하나의 AI 워크플로우

  • 보호자 발화 한 건이 상담·문진·예약·처방까지 끊김 없이 이어지도록 설계
  • 응급은 즉시 내원 유도, 일반은 RAG 문진으로 사전 정보를 모아 수의사에게 전달

Architecture
발화 한 건이 흐르는 길

보호자 메시지는 FastAPI가 SSE 스트림으로 받고, LangGraph 오케스트레이터가 매 턴 담당 에이전트를 결정합니다. 명확한 입력은 규칙으로, 애매한 자연어는 LLM이 분류합니다.

Deep Dive
가장 어려웠던 결정 — 누가 이 발화를 처리할 것인가

멀티에이전트의 핵심이자 가장 오래 붙잡은 문제는 라우팅이었습니다. “발화 한 건을 어느 에이전트에 보낼 것인가”를 잘못 정하면 그 뒤 문진·예약·처방이 전부 어긋납니다.

시도 1 · 순수 LLM 라우팅

왜 막혔나

  • 모든 발화를 LLM이 분류 → 비슷한 의도(예: “예약 바꿀래요” vs “예약 전에 물어볼게요”)에서 오분류 빈발
  • 턴마다 LLM 호출 → 지연·비용 증가, 디버깅도 “왜 이리 갔는지” 추적 어려움
시도 2 · 규칙+LLM 하이브리드 (채택)

어떻게 풀었나

  • 명확한 입력은 규칙으로 즉시 분기, 애매한 자연어만 LLM에 위임
  • 예약 전/후(phase)로 후보 에이전트를 제한 → 구조적으로 오분류 차단
  • SessionContext에 phase·slot·memo를 영속화해 멀티턴 맥락 유지
✅ 결과 — 약 1,000건 종합 QA에서 E2E 라우팅 정확도 100%, 평균 3.7턴으로 간결하게 대화 완료. 별도 검수에서 규칙·실제 처리 불일치 50건 중 실버그는 1건, 위험 카테고리 안전성 판정 fail 0건.

01 · MULTI-AGENT 설계 주도

멀티에이전트 오케스트레이션

  • 규칙 + LLM 하이브리드 라우팅 초기 설계 (명확=규칙 / 애매=LLM)
  • 예약 전/후(phase)로 후보 제한 → 오분류 방지
  • SessionContext로 멀티턴 상태 영속화
  • 후반 운영기에는 Follow-up Filter 에이전트를 단독으로 담당
02 · RAG 설계 주도

RAG 문진 + 할루시네이션 방어

  • text-embedding-3-small(1536d) + pgvector 초기 설계
  • 유사도 0.60 임계값 · LLM 쿼리 확장(fail-open)
  • 3중 방어: 임계값 + RAGAS + CI 게이트
03 · UX 기여

보호자 앱 UX · 응급 대화

  • 기계적 "병원 가세요" 대신 공감 → 즉시 내원 → 빠른 예약 제안
  • 안전 회귀(진단 단정·자가처치) 0건 유지
